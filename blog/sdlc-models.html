SDLC Models: How Each One Changes What Testing Looks Like

In the SDLC post I covered the phases every model shares in some form. What actually changes between models isn't the list of phases — it's how they're arranged in time, and that arrangement changes what a QA engineer's day-to-day actually looks like far more than any process document admits.

Waterfall: testing as a single late phase

Requirements, design, implementation, testing, deployment — each phase fully completes before the next begins. Testing happens once, at the end, against a fully built product. The upside is a clear, well-defined scope to test against. The downside is that any defect rooted in a requirement or design decision is now maximally expensive to fix, because everything downstream was already built on top of it. QA's real leverage in Waterfall is upfront — reviewing requirements and design carefully — precisely because there's no later cycle to catch what gets missed.

V-Model: testing planned in parallel, executed at the end

The V-Model pairs each development phase with a corresponding testing phase decided at the same time — acceptance tests planned alongside requirements, system tests planned alongside high-level design, and so on — even though execution still happens sequentially, similar to Waterfall. The real difference from plain Waterfall is that test planning isn't deferred; by the time implementation starts, what will eventually be tested is already defined. It reduces (but doesn't eliminate) the risk of testing scope being an afterthought.

Iterative: testing repeats every cycle

Instead of one pass through the full life cycle, the product gets built in repeated iterations, each producing a working (if incomplete) version. Testing happens every iteration, against a growing but real product, rather than once against a finished one. This surfaces integration problems far earlier, but it also means regression testing becomes a recurring cost from early on, not a once-at-the-end concern — a regression suite that doesn't scale becomes a bottleneck fast in this model.

Spiral: testing scoped by risk, every loop

Spiral adds an explicit risk-analysis step to each iteration, adjusting scope based on what's riskiest at that point in the project. For QA, this means test focus isn't fixed in advance — it shifts each cycle toward whatever's been flagged as the biggest current risk, which requires staying closely tied into planning rather than just receiving a fixed test plan.

Agile: testing embedded, not scheduled

Agile compresses the cycle into short sprints, each intended to ship potentially releasable, tested work. Testing isn't a phase that happens after a sprint's development — it happens continuously within it, ideally starting the moment a story is defined, running alongside implementation, and finishing before the sprint closes. This is the model where the parallel relationship between STLC and SDLC becomes most visible and most necessary — there's no separate testing phase left to fall back on if testing starts late.

The pattern across all five: the earlier and more frequently a model brings testing into the cycle, the earlier defects get caught and the cheaper they are to fix — but that benefit only shows up if testing actually keeps pace. An iterative or Agile project with testing that quietly falls a sprint behind gets the worst of both worlds: the complexity of constant change, without the early feedback that was supposed to justify it.

None of these models are right or wrong in the abstract — they fit different kinds of projects and different tolerances for changing requirements. But knowing which one a project is actually using changes something very practical: when in the timeline your test planning needs to start, and how often you should expect to repeat it.
