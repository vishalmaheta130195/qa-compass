Web App Testing Scenarios Worth Writing Down

Following on from test scenarios vs. test cases and why web apps need a different test approach — here's the actual scenario list I reach for on a typical web app, organized by the category of risk each one is checking for.

Authentication and session

  • Login with valid and invalid credentials, including a locked or disabled account.
  • Session behavior on refresh, on a second tab, and after the session naturally expires mid-action.
  • What happens if a logged-in user's permissions change while they're still in an active session.
  • Logout actually invalidating the session, not just redirecting to a login screen while the token still works.

Forms and input

  • Required fields left empty, and fields filled with the minimum and maximum allowed length.
  • Submitting the same form twice quickly — does it create a duplicate record, or is that prevented?
  • Special characters, emoji, and unexpected formats in text fields that assume plain text.
  • Navigating away mid-form and returning — is partially entered data preserved, lost, or does it silently submit something incomplete?

Navigation and state

  • Using the browser's back and forward buttons through a multi-step flow, not just the app's own navigation.
  • Deep-linking directly to a page that normally requires earlier steps first.
  • Opening the same resource in two tabs and making conflicting changes in each.

Data and persistence

  • Create, then immediately edit, then delete the same record — does each step correctly reflect the last?
  • What a list or table looks like with zero records, exactly one record, and enough records to require pagination.
  • Concurrent edits to the same record from two different sessions.

Errors and edge conditions

  • A slow or dropped network mid-request — does the UI hang, show a clear error, or silently fail?
  • Server error responses (not just the success path) — does the UI show something useful, or a raw error?
  • Clicking a submit button multiple times rapidly before the first request resolves.
Notice how few of these are about whether a feature technically works when used exactly as intended — almost all of them are about the boundary: refresh, back button, double submission, network hiccup, empty state. That's deliberate. The intended path is what gets built correctly by default; the boundaries are where the actual testing value is.

This isn't an exhaustive list for any specific app — it's a starting checklist to adapt. The categories (auth, forms, navigation, data, errors) are the ones that show up on nearly every web app regardless of what it does; the specific scenarios inside each one are worth reviewing against whatever you're actually testing, since the exact risks shift with what the feature does.
