Retesting vs. Regression Testing: Confirming the Fix Isn't the Same as Confirming Nothing Else Broke

These two happen back to back after almost every fix, which is exactly why they get treated as one step instead of two. They're answering different questions. Retesting asks: is the specific bug that was reported actually fixed. Regression testing asks: did fixing it break anything that used to work. Skipping the second one because the first one passed is one of the most common ways a "confirmed fixed" bug still ships a broken release.

Retesting: narrow, targeted, specific

Retesting means running the exact original failing steps — the same preconditions, same input, same sequence — against the fix, to confirm the specific reported behavior is now correct. It's narrow by design: it only tells you about the one issue it's checking, nothing about the rest of the system. I covered this in more depth in the bug life cycle post — it's the "Retest" stage specifically, and treating a "Fixed" status as unverified until this step has actually run against the original steps.

Regression testing: broad, exploratory, protective

Regression testing checks whether previously working functionality still works after a change — not the bug that was fixed, but everything around it that could plausibly have been affected. A fix that touches shared code, a common utility function, or a piece of state used elsewhere in the app can easily resolve the reported bug while quietly breaking something unrelated that nobody thought to check.

Why both are necessary, not redundant

A fix can pass retesting perfectly — the original bug is genuinely gone — while simultaneously breaking three other things regression testing would have caught. The two aren't sequential steps toward the same goal; they're two different questions that both need answering before a fix is actually safe to ship. Running only retesting after every fix is how a codebase slowly accumulates regressions that nobody explicitly signed off on, one small "safe" fix at a time.

A practical rule of thumb: the more shared or central the code being changed is, the more regression testing matters relative to retesting. A fix isolated to one rarely-used screen has a small blast radius. A fix inside a shared authentication function, a common date formatter, or anything else touched by many features deserves a regression pass proportional to how many things depend on it — not just a quick retest of the original bug.

What this looks like in practice

Retest first, using the original steps exactly as reported. Then run a regression pass scoped to what the fix actually touched — the affected module at minimum, and the broader smoke-test suite if the change was to shared or central code. Skipping straight from "the bug is gone" to "ship it" is exactly the gap where a fix quietly introduces a new problem nobody was looking for.
