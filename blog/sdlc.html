SDLC From a QA Seat, Not a Textbook

The Software Development Life Cycle gets taught as six or seven clean phases in a row. In real projects, they overlap constantly, and QA's actual involvement in each one looks very different from what the diagram suggests. Here's what each phase means once you're the person responsible for catching what slips through it.

Requirement Analysis: where most bugs are actually born

Requirements that are vague, contradictory, or simply incomplete cause far more defects than sloppy code does. If "the search should be fast" makes it into a requirements document without a number attached, that ambiguity turns into a dispute later — usually during testing, when it's most expensive to resolve. Reading requirements with a tester's eye, and flagging the vague ones before development starts, is some of the highest-leverage work QA can do, even though it happens before a single test case exists.

Design: the last cheap chance to ask "how would we test this?"

Once architecture decisions are locked in, testability is largely fixed too. A system with no logging, no way to inspect intermediate state, or no test environment that mirrors production will be painful to test no matter how skilled the QA team is. Getting a seat at design discussions — even just to ask what's observable and what isn't — pays off far more than any amount of test-case writing later.

Implementation: QA's quiet phase, if you use it well

While developers are building, this is when test plans, test data, and test scenarios actually get written — not after a build lands. Teams that wait until code is "done" to start thinking about testing are always testing under time pressure. Teams that use the implementation phase to prepare are testing on schedule.

Testing: not one phase, several kinds happening in sequence

This is usually where SDLC diagrams get lazy and just say "Testing" as a single box. In practice it's functional testing, regression testing, integration testing, and often performance and security testing layered on top of each other, each catching different classes of problems. Treating "Testing" as one undifferentiated phase is how teams end up thinking they're done after a single pass.

Deployment: the environment is part of the product

Code that passed every test in staging can still fail in production because of a configuration difference, a missing environment variable, or a permission that exists in one environment and not the other. Smoke-testing immediately after deployment — not just trusting that "it worked in staging" — is the difference between catching an environment-specific bug in minutes versus finding out from a user report.

Maintenance: where regression testing either saves you or doesn't

Every fix, patch, and small feature added after release risks breaking something that used to work. This is the phase where a solid regression suite pays for itself many times over — and where teams without one slowly accumulate quiet, unnoticed breakage until something visible finally fails.

The phases aren't really sequential in practice — they loop, overlap, and repeat every sprint or release cycle. What stays constant is that QA's job changes shape in each one: reading requirements critically, questioning design for testability, preparing early, testing in layers, verifying environments, and guarding against regression. It's less "the testing phase" and more "a different kind of attention in every phase."

If your team only associates QA with the "Testing" box on the diagram, that's usually a sign testing is starting later than it should — and picking up problems that were cheaper to catch two phases earlier.
